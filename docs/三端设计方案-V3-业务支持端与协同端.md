# 三端设计方案 V3 — 业务支持端 & 协同端

> 基于 630智能体项目进展汇报会议纪要（2026-05-18）
> 设计日期：2026-05-20

---

## 一、背景与上下文

### 项目目标
基于会议纪要核心指示，围绕"员工端 → 业务支持端 → 协同端"三端体系，在公司级员工工作台基础上，扩建**业务支持端**和**协同端**，形成完整的三端业务闭环。

### 三端定位
| 端 | 角色 | 用户 | 核心职责 |
|---|---|---|---|
| 员工端（已建） | 业务助理机器人 | 一线业务人员 | 通过AI团队完成日常业务 |
| 业务支持端（新建） | 各条线支持中心看板 | 质控/复核等支持岗位人员 | 统一汇聚入口，管业务动作，不管人 |
| 协同端（新建） | 圆桌会议跨部门协同 | 高级领导 | 以议题/项目为单元，协调跨部门资源和重大事项 |

### 重要设计原则
1. **三端不是三个隔离系统**，是同一平台上的三类角色视图，通过统一的事项/议题数据流连通
2. **不要线性串联，要平行可交叉**：三个闭环可并行触发，员工也可直接越级提请协同端
3. **业务支持中心不管人，只管业务和动作**：人员组织属于 HR 和团队长管理范畴
4. **以议题/项目为单元**处理协同端事务，而非以单个事项为单位

## 二、整体架构

### 数据流向图

```
员工端                    业务支持端                协同端
（已建）                   （新建）                  （新建）

[员工发起业务]             [支持中心看板]            [圆桌议题]
      │                        │                        │
      │ ←── 主动提交事项 ──→   │                        │
      │ ←── 规则触达/知会 ←──  │                        │
      │                        │ ←── 提请协同 ──────→  │
      │ ←────── 直接提请（跨部门/资源申请）──────────→ │
      │                        │ ←── 规则知会 ──────→  │
      │ ←──────────── 决策/指令下发 ←────────────────  │

                    ┌────────────────────────┐
                    │  AI 预判预处理层（贯穿三端）│
                    └────────────────────────┘
```

### 三个业务闭环

**闭环一：员工独立完成**（最常见）
```
员工发起业务 → AI智能体处理 → 员工确认/输出 → 完成
（支持端在后台自动感知，不介入处理）
```

**闭环二：员工 → 支持中心协同**（需质控/复核）
```
员工主动提交事项 → AI预判+分类路由 → 支持中心看板接单 → 决策/质控/复核 → 完成
（支持端也可主动前置介入，无需等员工提交）
```

**闭环三：支持中心 → 圆桌会议协同**（跨部门/资源申请）
```
支持端/员工提请协同 → 规则研判+议题生成 → 圆桌会议议题 → 高级领导决策 → 指令下发至相关端
```

## 三、业务支持端设计

### 3.1 定位

各条线（资管/投行/零售等）支持中心人员进入后看到**统一汇聚视图**，可以看到自己需要处理的工作。

- **不管人**：页面不按人员分区，不展示组织架构，只展示业务事项和动作
- **AI先行**：每条事项 AI 已完成预判，支持人员在预判基础上决策
- **主动介入**：支持人员无需等待员工提交，可根据规则监控主动前置介入

### 3.2 事项来源

| 来源类型 | 触发方式 | 进入队列 |
|---|---|---|
| 员工主动提交 | 员工在员工端点击"提交事项" | 待处理队列 |
| 规则自动触发 | 业务指标异常/阈值触发 | 关注队列（知会，暂不处理） |
| 支持人员主动发现 | 支持人员在监控中主动介入 | 处理中队列 |

### 3.3 AI 预处理规则

- **数据完整度检测**：识别事项中缺失的关键数据（资产负债/现金流等核心维度）
- **异常识别**：基于客观指标检测异常（不使用主观评价）
- **历史案例匹配**：相似案例相似度匹配
- **优先级自动定级**：紧急 / 普通 / 关注（规则自动判定）
- **建议动作输出**：直接处理 / 补录数据 / 提请协同

### 3.4 页面功能模块

1. **概览看板**：今日待处理/关注/已完成统计，按条线分区
2. **工作队列侧边**：左侧显示各条线队列及数量，可按条线切换
3. **事项列表**：展示 AI 预判结果+建议动作，支持"处理/查看/提请协同/知晓/提前介入"等操作
4. **规则知会区**：系统自动触发的关注事项单独展示，支持"知晓/主动介入"两种响应

### 3.5 与员工端的连接点

- 员工端：点击"提交事项"→ 填写结构化事项单 → 进入支持端待处理队列
- 支持端：处理完毕 → 结果回传员工端（状态更新/补充意见）
- 支持端：无法处理 → 一键"提请协同"→ 生成协同端议题

## 四、协同端设计

### 4.1 定位

以**议题/项目**为处理单元，供高级领导主导跨部门协调，处理两端无法独立解决的重大事项。

- **不是日常工作台**：协同端处理的是例外情况，不是常规流程
- **领导主导**：参与者是各条线高级领导及职能部门负责人
- **决策闭环**：决策完成后以结构化指令形式下发，并追踪执行状态

### 4.2 议题触发规则

| 触发方式 | 说明 |
|---|---|
| 主动发起 | 员工端/支持端人员点击"提请协同"，填写结构化申请单，系统生成议题 |
| 规则自动知会 | 重大事项阈值（金额超限、风险等级、多部门冲突）触发，自动推送至协同端（知晓类，不一定需要处理） |
| 事项升级 | 支持端无法处理的事项，一键升级为协同端议题 |

### 4.3 议题类型

| 类型 | 说明 | 示例 |
|---|---|---|
| 跨部门客户冲突 | 多条线共同客户策略不一致 | 资管投行双条线客户口径统一 |
| 资源申请 | 算力、预算、人员跨部门调配 | 630节点算力扩充申请 |
| 数据权限 | 跨条线数据访问权限授权 | 零售智能体数据权限申请 |
| 重大风险知会 | AI监控发现的重大异常知会领导 | IPO项目连续异常指标监控 |
| 制度/机制决策 | 需高层拍板的制度或机制调整 | 薪酬奖金方案、组织协同规则 |

### 4.4 协同端核心流程

```
议题生成 → 参与方召集（动态邀请涉及部门）→ 圆桌讨论
→ 高级领导决策 → 形成决策结论+结构化指令 → 下发至相关端
→ 执行追踪 → 未按期完成自动升级提醒 → 关闭议题
```

### 4.5 页面功能模块

1. **议题列表**：展示当前所有议题，按优先级/状态/类型筛选
2. **议题详情**：包含背景、涉及部门、关联事项、讨论记录、决策结论
3. **知会区**：规则自动推送的知会类议题，领导知晓即可，支持一键回复"已知晓"
4. **决策下发**：高级领导做出决策后，填写结构化指令，一键下发至指定部门/人员
5. **执行追踪看板**：已下发指令的执行状态追踪，逾期自动标红

## 五、核心数据模型

### 5.1 事项（WorkItem）—— 员工端 ↔ 业务支持端

| 字段 | 类型 | 说明 |
|---|---|---|
| item_id | UUID | 唯一标识 |
| source_type | Enum | 员工主动提交 / 规则触发 |
| business_line | Enum | 资管 / 投行 / 零售 |
| business_stage | Enum | 客户分析 / 业务分析 / 方案生成 / 客户服务 |
| subject | Object | **对象-动作-结果**（三层无歧义描述） |
| ai_prejudge | Object | AI预判结论 + 建议动作 |
| status | Enum | 待处理 / 处理中 / 已完成 / 已升级 |
| priority | Enum | 紧急 / 普通 / 关注（规则自动定级） |
| escalate_to_agenda | UUID? | 关联的协同端议题ID（如有升级） |
| created_at | DateTime | 创建时间 |
| resolved_at | DateTime? | 处理完成时间 |

**subject 三层无歧义结构（遵循陶总指示）**：
```json
{
  "object": "华创证券资管条线-刘韵 与 中信银行贵阳分行-财务总监王某",
  "action": "提交债券投资方案复核",
  "result": "方案现金流数据缺失，待补录后重新提交"
}
```

### 5.2 议题（Agenda）—— 业务支持端 ↔ 协同端

| 字段 | 类型 | 说明 |
|---|---|---|
| agenda_id | UUID | 唯一标识 |
| trigger_type | Enum | 主动发起 / 规则知会 / 事项升级 |
| topic | String | 议题标题 |
| agenda_type | Enum | 跨部门冲突 / 资源申请 / 数据权限 / 风险知会 / 制度决策 |
| dept_involved | Array | 涉及部门列表 |
| coordinators | Array | 参与协调的高级领导 |
| resource_type | Enum | 算力 / 预算 / 数据权限 / 人员 / 跨部门 |
| decision | Object | 决策内容 + 指令 |
| instructions | Array | 下发给各端的具体结构化指令 |
| tracking_status | Enum | 待讨论 / 决策中 / 已下发 / 执行追踪 / 关闭 |
| related_items | Array | 关联的事项ID列表 |
| priority | Enum | 高优 / 中优 / 知会 |
| created_at | DateTime | 创建时间 |

### 5.3 状态流转

```
【事项状态机】
员工提交/规则触发 → AI预处理 → 待处理 → 处理中 → 已完成
                                    ↓（需协同）
                                  已升级 ────→ 生成议题

【议题状态机】
议题生成 → 待讨论 → 决策中 → 已下发 → 执行追踪 → 关闭
```

## 六、建设规划

### 阶段划分

**近期（630节点前）**
- 业务支持端 MVP：事项队列 + AI预处理 + 提请协同入口
- 协同端 MVP：议题列表 + 知会展示 + 决策下发

**中期**
- 三端数据互通：事项状态实时同步、指令下发追踪
- 各条线支持中心独立视图配置
- 规则引擎配置界面（业务支持端触发规则可配置）

**远期**
- 跨条线数据分析：支持端处理效率分析、事项类型分析
- 协同端决策历史库：形成公司级决策知识沉淀
- 对外输出能力：投行/资管服务能力平台化输出

## 七、关键设计决策记录

| 决策点 | 决策 | 原因 |
|---|---|---|
| "工单"改为"事项" | 统一使用"事项"表述 | 符合公司业务语言习惯，避免系统感过强 |
| 支持端不管人 | 只展示业务事项和动作，不展示人员组织 | 陶总明确指示：人员组织属于HR和团队长管理 |
| 协同端以议题为单元 | 不以单条事项为单位 | 协同端处理的是聚合的跨部门问题，非原子事项 |
| 三端可并行触发 | 员工可直接越级提请协同端 | 不强制串联，避免流程瓶颈 |
| AI预判前置 | 所有事项进入支持端前AI已完成预处理 | 减少支持人员重复判断工作，聚焦决策 |
| subject三层无歧义 | 对象-动作-结果结构化描述 | 陶总明确：大模型无法处理模糊表述 |
